# NDAX Windows Guide

Этот репозиторий обрабатывает `.ndax` файлы двумя поддерживаемыми командами:

- `.\plot_ndax.cmd` строит JPG-график по одному или нескольким входным файлам.
- `.\build_comparison_table.cmd` строит CSV-таблицу с экстремумами и временем короткого замыкания.

## Требования

- Windows PowerShell или `cmd.exe`
- Python >= 3.12
- Git, если вы клонируете репозиторий через `git clone`

## Подготовка проекта на Windows

1. Получите проект.

Если у вас уже установлен `git`, выполните:

```powershell
git clone https://github.com/egopbi/ndax_processing_scripts.git
cd ndax_processing_scripts
```

Если `git` не установлен, скачайте репозиторий ZIP-архивом, распакуйте его и перейдите в папку проекта:

```powershell
cd C:\путь\к\ndax_processing_scripts
```

2. Установите Python >= 3.12 с официальной страницы для Windows и включите опцию `Add python.exe to PATH`.
3. Проверьте, что Python доступен из терминала:

```powershell
py --version
```

Ожидается версия `3.12` или выше. Если команда `py` не найдена, закройте и заново откройте PowerShell. Если это не помогло, переустановите Python и убедитесь, что включена опция `Add python.exe to PATH`.

4. Если вы открыли PowerShell не сразу в корне проекта, перейдите в папку репозитория:

```powershell
cd C:\путь\к\ndax_processing_scripts
```

5. Выполните:

```powershell
.\setup_windows.cmd
```

`.\setup_windows.cmd` создает локальное окружение `.venv`, обновляет `pip` и устанавливает зависимости. Обычно этого достаточно один раз на машину или после обновления зависимостей.

Если PowerShell не разрешает запуск локального `.cmd`, используйте:

```powershell
cmd /c setup_windows.cmd
```

## Основные команды

Построить график:

```powershell
.\plot_ndax.cmd --files examples\example1_1.ndax --y-column Voltage
```

Построить сравнительную таблицу:

```powershell
.\build_comparison_table.cmd --files examples\example1_1.ndax --y-column Voltage --anchor-x 0.5
```

## Аргументы CLI

`.\plot_ndax.cmd` принимает:

- `--files` один или несколько путей к `.ndax`
- `--y-column` имя столбца для оси Y
- `--x-column` имя столбца для оси X, по умолчанию `Time`
- `--labels` подписи серий
- `--x-min`, `--x-max`, `--y-min`, `--y-max` числовые пределы осей
- `--output` путь к итоговому JPG

`.\build_comparison_table.cmd` принимает:

- `--files` один или несколько путей к `.ndax`
- `--y-column` имя столбца для анализа
- `--anchor-x` одна или несколько опорных точек по оси X
- `--x-column` имя столбца для оси X, по умолчанию `Time`
- `--labels` подписи строк
- `--output` путь к итоговому CSV

По умолчанию артефакты сохраняются в папку `output`.

Если `--labels` не указан:

- названия строк берутся из имён файлов без расширения.

Перед обычным запуском этой команды достаточно один раз выполнить `.\setup_windows.cmd`. После этого вручную активировать `.venv` не нужно.

## Вспомогательные команды

Эти команды есть в проекте, но обычно они не нужны для повседневной работы.

### Проверка файла

Эта команда нужна в основном для диагностики, а не для обычной работы.

```powershell
.\.venv\Scripts\python.exe scripts\health_check_ndax.py examples\example1_1.ndax
```

## Как рассчитываются 6 точек таблицы

Ниже описана логика без кода.

1. Сначала скрипт берёт одно или несколько значений `anchor-x` как опорные точки по оси X.
2. Для каждого `anchor-x` отдельно он ищет в данных ближайшую реальную точку X к этому значению.
3. Затем он строит вспомогательный ряд `y_det`. Это не новый физический сигнал, а сглаженная версия исходного ряда, которая помогает видеть общую форму сигнала, а не мелкую рябь.
4. По `y_det` скрипт делит участок вокруг `anchor-x` на положительные, отрицательные и нейтральные зоны.
5. Из всех найденных пар `положительная зона -> отрицательная зона` он выбирает ближайшую по времени к `anchor-x`.
6. В выбранной положительной зоне он ищет тройку `максимум -> минимум -> максимум`, а в выбранной отрицательной зоне тройку `минимум -> максимум -> минимум`.
7. В итоговую таблицу попадают индексы, найденные алгоритмом, но сами значения берутся из исходного сигнала, а не из `y_det`.
8. Для каждого `anchor-x` создаётся свой блок точек `+U_l`, `+U_m`, `+U_r`, `-U_l`, `-U_m`, `-U_r`.

Важно понимать:

- `anchor-x` не обязан попадать точно между `U+` и `U-`; он задаёт только примерную область поиска;
- алгоритм знает, что точки `U+` должны быть в положительной зоне, а точки `U-` в отрицательной зоне;
- `y_det` считается по окну соседних точек, поэтому одиночные случайные зубцы меньше влияют на поиск;
- если какая-то точка не нашлась, соответствующая ячейка остаётся пустой;
- перед поиском убираются начальные строки со статусом `Rest`;
- если ось X равна `Time`, для поиска используется время в секундах внутри данных, но в командной строке каждое значение `--anchor-x` вы всё равно задаёте в часах.

## Как определяется точка короткого замыкания

Скрипт использует два независимых признака и берёт более ранний из них.

Первый признак: пороговое событие.

- после предварительной очистки данных ищется первый локальный максимум не ниже `200` мВ;
- или первый локальный минимум не выше `-200` мВ;
- если такой экстремум найден, его время считается кандидатом на короткое замыкание.

Второй признак: схлопывание амплитуды сигнала.

- сигнал делится на окна по 5 часов;
- для каждого окна считается амплитуда как разница между максимумом и минимумом;
- базовый уровень берётся как медиана трёх предыдущих окон;
- короткое замыкание фиксируется, если через 5 часов амплитуда уже меньше 80% базы, через 10 часов меньше 50% базы, и дальше не восстанавливается выше 60% базы.

Что ещё важно:

- перед этим анализом убираются начальные строки `Rest`;
- если в данных напряжения есть длинный стартовый хвост до первого экстремума, он тоже отрезается;
- если в файле есть корректный столбец `Timestamp`, время считается по нему, иначе берётся столбец `Time`;
- итоговое значение в таблице округляется до ближайших 5 часов по правилу half-up: например, `12.5` станет `15`, а `12.4` станет `10`.

## Настройка project_config.yaml

Файл `project_config.yaml` управляет общими настройками проекта. Меняйте его только если понимаете, зачем именно это нужно.

### `paths.output_dir`

Что меняет:
папку, куда по умолчанию сохраняются графики и таблицы.

Какое значение ждёт:
строку, например `output`.

Когда имеет смысл менять:
если вы хотите складывать все результаты в другую папку.

Когда лучше не менять:
если вас устраивает папка `output` и вы не хотите перестраивать привычный путь к результатам.

### `plot.palette`

Что меняет:
цвета линий на графике.

Какое значение ждёт:
список строк с цветами, например в формате `#1718FE`.

Когда имеет смысл менять:
если вам нужны другие фирменные цвета или более контрастные линии.

Когда лучше не менять:
если вы не уверены, что новых цветов хватит на нужное количество файлов. Сейчас палитра рассчитана на 8 линий; если файлов больше, график не построится.

### `plot.defaults.x_column`

Что меняет:
стандартное имя X-столбца для `health_check_ndax.py`, где это значение берётся из конфигурации как обязательный столбец для графика.

Какое значение ждёт:
строку, сейчас это `Time`.

Когда имеет смысл менять:
почти никогда для обычной работы. Основная команда `plot_ndax.py` всё равно берёт X-столбец из флага `--x-column` или использует своё встроенное значение `Time`.

Когда лучше не менять:
если вы пользуетесь основными командами и не хотите ломать демонстрационные сценарии.

### `plot.defaults.y_column`

Что меняет:
стандартное имя Y-столбца для `health_check_ndax.py`, где это значение берётся из конфигурации как обязательный столбец для графика.

Какое значение ждёт:
строку, сейчас это `Voltage`.

Когда имеет смысл менять:
только если вы осознанно меняете столбец, который health check считает обязательным для графика.

Когда лучше не менять:
если вы строите обычные графики через `plot_ndax.py`, где нужный столбец удобнее задавать флагом `--y-column`.

### `csv.defaults.columns`

Что меняет:
список CSV-столбцов, который использует `health_check_ndax.py`.

Какое значение ждёт:
список строк. Сейчас это:

- `Time`
- `Voltage`
- `Current(mA)`
- `Charge_Capacity(mAh)`
- `Discharge_Capacity(mAh)`

Когда имеет смысл менять:
если вы хотите, чтобы проверка файла ориентировалась на другой набор обязательных столбцов.

Когда лучше не менять:
если вы не готовы одновременно менять свои ожидания от health check.

### `comparison_table.extrema_detection.window_points`

Что меняет:
сколько соседних точек алгоритм берёт, когда строит вспомогательный ряд `y_det` для поиска 6 ключевых точек.

Какое значение ждёт:
нечётное целое число не меньше `3`. Сейчас по умолчанию используется `9`.

Когда имеет смысл менять:
если алгоритм слишком чувствителен к мелким колебаниям и цепляется за шум. Тогда окно можно увеличить.

Когда лучше не менять:
если таблица уже находит нужные крупные пики и впадины. Слишком большое окно может чрезмерно сгладить форму сигнала.

### `comparison_table.extrema_detection.zero_threshold`

Что меняет:
насколько близкие к нулю значения считать нейтральной зоной, а не уверенно положительной или отрицательной.

Какое значение ждёт:
неотрицательное число. Сейчас по умолчанию используется `5.0`.

Когда имеет смысл менять:
если около нуля сигнал дрожит и алгоритм слишком часто переключается между `U+` и `U-` зонами.

Когда лучше не менять:
если положительные и отрицательные части сигнала уже отделяются друг от друга уверенно.

### `comparison_table.extrema_detection.min_zone_points`

Что меняет:
минимальную длину непрерывной положительной или отрицательной зоны, которую алгоритм считает настоящей частью сигнала, а не шумом.

Какое значение ждёт:
положительное целое число. Сейчас по умолчанию используется `5`.

Когда имеет смысл менять:
если в таблицу попадают короткие случайные куски сигнала рядом с нулём.

Когда лучше не менять:
если алгоритм уже стабильно выбирает нужные зоны. Слишком большое значение может отбросить короткий, но реальный участок сигнала.

### `comparison_table.extrema_detection.min_extrema_separation_points`

Что меняет:
минимальное расстояние в точках между соседними ключевыми экстремумами внутри одной тройки.

Какое значение ждёт:
положительное целое число. Сейчас по умолчанию используется `5`.

Когда имеет смысл менять:
если алгоритм выбирает почти соседние точки вместо заметного пика и впадины.

Когда лучше не менять:
если нужные экстремумы уже находятся правильно. Слишком большое значение может сделать поиск слишком строгим и оставить часть ячеек пустыми.

## Частые вопросы и ошибки

### Команда пишет, что столбец не найден

Проверьте точное имя столбца внутри файла NDAX. Поиск не чувствителен к регистру, но имя должно совпасть однозначно.

### Команда пишет, что количество `--labels` не совпадает

Подписей должно быть ровно столько же, сколько файлов в `--files`.

### После `--x-min` / `--x-max` или `--y-min` / `--y-max` график пустой

Значит, выбранный диапазон отрезал все точки. Уберите ограничения или задайте более широкий интервал.

### Почему график или таблица начинаются не с самого первого значения

Перед обработкой проект убирает начальные строки со статусом `Rest`. Для графика и поиска короткого замыкания также может быть отрезан длинный стартовый хвост до первого выраженного экстремума.

### Почему в таблице пустые ячейки

Это означает, что для одной из шести точек алгоритм не нашёл устойчивую положительную или отрицательную зону, либо внутри выбранной зоны не нашлась подходящая тройка экстремумов.

### Почему health check ничего не сохраняет в папку

Так и задумано. Эта команда только печатает отчёт в терминал и завершает работу с кодом `0` или `1`.
