# README User Guide Design

**Date:** 2026-03-29

## Goal

Переписать `README.md` как понятную пользовательскую инструкцию для человека без технического опыта. README должен объяснять, что делает проект, как подготовить среду для запуска, какие команды доступны, какие флаги и значения они принимают, как интерпретировать результаты, и как менять настройки в `project_config.yaml`.

## Audience

- Основной читатель: пользователь лабораторного сценария, не знакомый с Python, терминалом и внутренним устройством проекта.
- Вторичный читатель: технический коллега, которому нужен быстрый справочник по командам и флагам.

## Information Architecture

README должен идти от простого к подробному:

1. Кратко: что делает проект.
2. Подготовка проекта к запуску.
3. Список основных команд.
4. Короткие базовые примеры.
5. Подробный справочник по каждой команде.
6. Логика работы графика и таблицы.
7. Описание настроек `project_config.yaml`.
8. Полезные замечания и частые ошибки.

Это сохраняет низкий порог входа в начале файла и переносит алгоритмические детали ниже, где их будут читать только заинтересованные пользователи.

## Commands To Cover

Основные пользовательские команды:

- `scripts/plot_ndax.py`
- `scripts/build_comparison_table.py`
- `scripts/health_check_ndax.py`

Вспомогательные команды:

- `scripts/generate_plot_poc.py`
- `scripts/generate_csv_poc.py`

README должен явно разделять основные и вспомогательные команды, чтобы пользователь не путал рабочий интерфейс с демонстрационными скриптами.

## Accuracy Requirements

README должен опираться на текущее состояние кода:

- реальные `argparse`-флаги и их обязательность
- реальные единицы измерения в публичном CLI
- реальное поведение `Time` и `Voltage`
- текущие поля `project_config.yaml`
- фактический алгоритм поиска шести точек
- фактический алгоритм определения короткого замыкания

Нельзя описывать удаленные настройки или устаревшие конфигурационные поля.

## Content Requirements

README должен включать:

- простое объяснение назначения проекта
- подготовку проекта для macOS и Windows
- объяснение, где запускать команды и какой файл подавать на вход
- таблицу или маркированный список доступных команд
- подробное описание обязательных и необязательных флагов
- допустимые типы значений для каждого флага
- несколько практических примеров для каждой основной команды
- объяснение, какие колонки можно использовать для осей и таблицы
- объяснение логики `anchor-x`
- объяснение шести точек `+U_l`, `+U_m`, `+U_r`, `-U_l`, `-U_m`, `-U_r`
- объяснение, как ищется точка короткого замыкания и как она округляется
- описание всех актуальных YAML-настроек и безопасных сценариев их изменения

## Scope Boundaries

В рамках этой задачи меняется только документация.

Не входит в scope:

- изменение Python-кода
- изменение CLI-контрактов
- добавление новых флагов
- перестройка структуры проекта

## Verification

Документация считается готовой, если:

- README соответствует текущим `--help` и коду
- порядок разделов отражает agreed structure: сначала подготовка и список команд, затем детали
- отсутствуют ссылки на удаленный `labels.axis_overrides`
- YAML-описание соответствует текущему `project_config.yaml`
- текст остается понятным нетехническому пользователю и не перегружен в верхней части
